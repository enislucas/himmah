Himmah · v6.10 · 🎥 Meet

▲ Google Meet links

How to connect Himmah to your Google Calendar so the 🎥 Meet link button on a calendar event creates a real Google Meet URL. Every step, with what you should see and what to do if you don't. Tick the boxes - they're remembered on this PC.

Before you start, know this: This is completely optional and only affects one button - the 🎥 on a calendar event. Everything else in Himmah works without it, and you can always paste a Meet link you already have with the 🔗 button instead - no setup needed. This is the longest setup in Himmah: about 10 minutes, one time, because Google's own permissions screens are fiddly (and Google reorganised them again recently - the steps below match what's live as of mid-2026). It's free (the Google Calendar API has no cost at this scale). You'll need a Google account. We use Google's own OAuth Playground so you never have to write or run any code.

Part 1 - A Google project with the Calendar API on (3 min)

1. Create a Google Cloud project Go to console.cloud.google.com/projectcreate, name it Himmah, and click Create. Wait a few seconds, then make sure it's the selected project (top bar).
You should see: "Himmah" shown as the current project in the blue top bar.
If it asks you to pick an organisation: choose "No organisation" - a personal project is fine.
2. Enable the Google Calendar API Go to this API Library page (Google Calendar API) and click Enable.
You should see: the Calendar API page now shows a "Manage" button and "API Enabled".
If "Enable" is greyed out or the page looks unrelated: the wrong project is selected - switch to "Himmah" in the top bar and reload the page.

Part 2 - OAuth credentials (4 min)

Google reorganised this area into Google Auth Platform (three tabs: Branding, Audience, Clients) - it used to be one page called "OAuth consent screen". If your console still shows the old single-page layout, the same fields exist there under different tab names; just follow the field names below rather than the exact tab.

3. Set the app name (Branding tab) Go to APIs & Services → Google Auth Platform → Branding. Enter an app name (Himmah) and a support email (your own), then Save.
You should see: the Branding page shows "Himmah" as the app name with no red warnings.
4. Set External + add yourself as a test user (Audience tab) Go to Google Auth Platform → Audience. Set the user type to External. Under Test users, click Add users and add your own Google email. You do not need to publish or get verified - leaving Publishing status as "Testing" is correct and normal for personal use.
You should see: your email listed under Test users, and Publishing status reads "Testing".
If you skip adding yourself as a test user: step 7 will fail with "access_denied" / "app not verified". Come back here and add your exact email.
5. Create an OAuth client ID (Web application) Go to Google Auth Platform → Clients (older consoles call this APIs & Services → Credentials - same feature). Click Create Client, Application type Web application. Under Authorised redirect URIs click "Add URI" and paste exactly: 
https://developers.google.com/oauthplayground
Click Create.
You should see: a popup with your Client ID (ends in .apps.googleusercontent.com) and Client secret (starts GOCSPX-). Copy both into a temporary note.
If you used "Desktop app" by mistake: the Playground won't accept it - delete it and create a Web application client with the exact redirect URI above (no trailing slash).

Part 3 - Get a refresh token with the OAuth Playground (2 min)

6. Tell the Playground to use your client Open developers.google.com/oauthplayground. Click the gear icon (top right) → tick "Use your own OAuth credentials" → paste your Client ID and Client secret from step 5.
You should see: the gear panel now shows your own Client ID instead of Google's shared one.
7. Choose the Calendar scope and authorise In the left Step 1 box, scroll to the empty "Input your own scopes" field and paste: 
https://www.googleapis.com/auth/calendar.events
Click Authorise APIs, sign in with the same Google account you added as a test user, and click Allow.
You should see: a Google warning "Google hasn't verified this app" - click Advanced → "Go to Himmah (unsafe)" → Allow. This is expected and normal for a personal, unverified app. You then return to the Playground on Step 2.
If you get "access_denied": the signed-in account isn't listed as a test user. Go back to Part 2, step 4 and add that exact email.
8. Exchange for tokens and copy the refresh token On Step 2, click Exchange authorisation code for tokens. A Refresh token appears (starts with 1//). Copy it.
You should see: a Refresh token field filled with a long 1//... string (an access token also appears - you can ignore it, Himmah gets its own from the refresh token automatically).
If the refresh token is blank: revoke access at myaccount.google.com/permissions (remove "Himmah"), then redo steps 7-8 - Google only hands out a refresh token on the first authorisation for a given app+account pair.

Part 4 - Give it to Himmah + test (1 min)

9. Fill in google_meet.json Open your Himmah folder → data → google_meet.json in Notepad. Replace its contents with this, pasting your three values (keep the quotes and commas exactly): 
{
  "client_id": "YOUR_CLIENT_ID.apps.googleusercontent.com",
  "client_secret": "GOCSPX-your_secret",
  "refresh_token": "1//your_refresh_token",
  "calendar_id": "primary"
}
Save. (Leave calendar_id as "primary" unless you want a specific calendar.)
You should see: a small valid JSON file - four keys, each in double quotes, no trailing comma after the last line.
If Himmah later says it can't read it: a stray comma or a "smart quote" from Notepad broke the JSON. Make sure every quote is a straight " character.
10. The real test: make a Meet link Reload Himmah (Ctrl+R). Open the Calendar, create or open an event, and in its description area click 🎥 Meet link.
You should see: a real https://meet.google.com/xxx-xxxx-xxx link dropped into the event's description, and the event now exists for real in your Google Calendar (not just in Himmah).
"Google isn't connected yet - open GOOGLE-MEET-SETUP-GUIDE.html to connect it.": ignore the filename in that message - it refers to Himmah's main build; in this share build, this guide is the setup doc. It means one of the three fields in google_meet.json is missing - re-check step 9. "Google API HTTP 401 - token may be expired; reconnect.": see the note below about 7-day tokens. If you don't want any of this: use the 🔗 button instead to paste a Meet link you already have - zero setup.
A real Google limit to know about: while your Google Cloud project's Audience status stays "Testing" (which is the normal, correct choice above), Google automatically expires the authorisation after about 7 days - this is a Google policy for unverified test apps, not a Himmah bug. If the Meet button starts failing with "Google API HTTP 401 - token may be expired; reconnect." after roughly a week of not using it, it's this expiry. The fix is always the same: redo Part 3 (steps 6-8) to get a fresh refresh token, paste it into google_meet.json, save, reload. (Advanced option: in Google Auth Platform → Audience you can click Publish app to move to "In production", which removes the 7-day limit - you'll still see the same "unverified app" warning to click through each time you re-authorise, since Calendar access is a "sensitive" scope Google doesn't require full review for at personal-use scale, but this is optional and not required for the guide above to work.)
